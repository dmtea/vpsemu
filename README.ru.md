# vpsemu

Скилл для AI-агента (opencode и совместимых CLI-агентов) который позволяет создавать,
управлять и удалять LXD-контейнеры имитирующие чистые VPS-серверы — для тестирования
Ansible-плейбуков и IaC-инфраструктуры.

---

## Что это и зачем

При разработке Ansible/IaC проектов нужна среда где можно быстро поднять чистый сервер,
прогнать плейбук, убедиться что всё работает — и сбросить обратно. Хостинг VPS для этого
дорого и медленно. Docker не подходит — systemd, SSH, сетевые интерфейсы там не такие
как на реальном сервере.

LXD-контейнеры решают это: запускаются за секунды, ведут себя как настоящий VPS
(полноценный systemd, SSH, статичный IP, root-доступ), и стоят 0 денег.

Скилл даёт агенту готовый набор команд + правила изоляции чтобы несколько агентов
могли работать параллельно не мешая друг другу.

---

## Структура папки

```
vpsemu/
├── vpsemu.skill.md      — сам скилл, подключается к агенту
├── vpsemu.skill.ru.md   — русская версия для чтения/редактирования
├── README.md            — документация (английский)
└── README.ru.md         — этот файл
```

---

## Требования

- Ubuntu 24.04 (хост)
- `snapd` для установки LXD
- `sops` если используется зашифрованный secrets-файл
- `python3`, `python3-yaml` на хосте (для парсинга SOPS-вывода)
- Проект с Ansible: ожидается структура `ansible/group_vars/all/secrets.enc.yml`

---

## Быстрый старт

1. Выполнить Bootstrap-секцию из `vpsemu.skill.md` — один раз на хосте
2. Подключить `vpsemu.skill.md` к агенту (через skills-папку или system prompt)
3. Агент при загрузке скилла автоматически генерирует `AGENT_ID` и готов к работе

---

## Подключение к агенту

Положить `vpsemu.skill.md` в папку скиллов агента. Для opencode и совместимых агентов
это обычно `~/.config/<agent>/skills/` или папка проекта `skills/`.

При загрузке скилла агент обязан немедленно выполнить инициализацию `AGENT_ID` —
это описано в секции `⚡ Инициализация` внутри скилла.

---

## Неймспейс контейнеров

```
vpsemu-{AGENT_ID}-{TASK_NAME}

Примеры:
  vpsemu-a3f9-deploy-nginx
  vpsemu-b7c1-setup-db
```

Каждый агент работает только со своими контейнерами. Параллельные агенты и задачи
не пересекаются ни по именам ни по IP-адресам.

## Changelog

### v0.01 — начальная версия
Базовая концепция: LXD как среда для обучения Ansible и IaC.
- Образ: `ubuntu:22.04`
- Одиночный контейнер, ручная настройка SSH, доступ по паролю root
- Изолированная сеть `lxdbr1` без DHCP, статичный IP задаётся вручную
- Базовый профиль `vps-base`, эталонный контейнер `vps-template` со снапшотом `clean-vps`
- Команды: создать / SSH / стоп / старт / удалить / снапшот / restore
- Без изоляции между агентами — один пользователь, одна задача

### v0.02 — мульти-агентность через SESSION_ID
- Введён неймспейс `{SESSION_ID}-{NAME}` для изоляции контейнеров между агентами
- `SESSION_ID` генерируется при старте сессии через `openssl rand -hex 3`
- Добавлена детерминированная IP-аллокация через MD5-хеш от имени контейнера
  (диапазон `10.10.0.10–249`) — исключает коллизии при параллельной работе
- Добавлены команды: список своих контейнеров, удалить все свои
- Добавлены правила безопасности: агент не трогает контейнеры с чужим префиксом

### v0.03 — переход на Ubuntu 24.04
- Обновлён базовый образ с `ubuntu:22.04` на `ubuntu:24.04`
- Добавлен `python3-apt` в зависимости шаблона (нужен для Ansible на 24.04)

### v0.10 — трёхуровневый неймспейс AGENT_ID + SESSION_ID + ROLE
- Неймспейс расширен до `{AGENT_ID}-{SESSION_ID}-{ROLE}` для поддержки
  нескольких агентов, нескольких задач у одного агента и нескольких контейнеров на задачу
- Добавлен статичный префикс `vpsemu-` для глобальной идентификации контейнеров скилла
- `AGENT_ID` — стабильный идентификатор агента, задаётся при инициализации
- `SESSION_ID` — идентификатор сессии, генерируется при каждом старте
- `ROLE` — роль контейнера в задаче (web, db, cache...)
- Расширены правила безопасности: агент не трогает контейнеры с чужим `AGENT_ID`
- Добавлен пример Python SDK (`pylxd`) для программного управления из агента

### v0.20 — упрощение: SESSION_ID убран, ROLE → TASK_NAME
- `SESSION_ID` убран как избыточный — дублировал смысл `AGENT_ID`
- `ROLE` переименован в `TASK_NAME` — точнее отражает назначение
- Итоговый неймспейс: `vpsemu-{AGENT_ID}-{TASK_NAME}`
- `AGENT_ID` — 4-hex, генерируется через `openssl rand -hex 2` при старте сессии
- Скилл переименован из `lxd-vps` в `vpsemu` (имя + файл)
- Секция "Разовая настройка хоста" переименована в `Bootstrap (выполняется владельцем...)`
  для явного разграничения: что делает человек, а что агент

### v0.21 — уточнение хранения AGENT_ID
- Убрана попытка хранить `AGENT_ID` в файле `~/.vpsemu_agent_id`
- Причина: с одной папкой проекта могут работать разные агенты одновременно —
  файл на диске привёл бы к коллизиям между ними
- `AGENT_ID` живёт исключительно в контексте сессии агента
- Добавлено явное требование: при сжатии контекста агент восстанавливает `AGENT_ID`
  из памяти, не генерирует заново

### v0.30 — интеграция SOPS для ROOT_PASS
- Убран хардкод `ROOT_PASS="rootpass"`
- Добавлена функция `_get_root_pass` с трёхуровневым fallback:
  1. читать `vault_root_password` из `ansible/group_vars/all/secrets.enc.yml` через `sops -d`
  2. файл есть, ключа нет → предупредить пользователя, спросить пароль в диалоге
  3. файла нет → спросить пользователя: другой путь или пароль напрямую
- Добавлена переменная `PROJECT_ROOT` через `git rev-parse --show-toplevel`
- Добавлено правило безопасности #6: `ROOT_PASS` не логируется, не пишется на диск,
  при fallback передаётся только через `-e` в момент вызова ansible
- Убран `pass=${ROOT_PASS}` из финального `echo "READY..."` (утечка в лог)

### v0.31 — исправление логики пароля в шаблоне
- Обнаружен архитектурный баг: пароль устанавливался в bootstrap хардкодом,
  наследовался снапшотом, и не совпадал с реальным паролем из SOPS
- Исправление: пароль полностью убран из шаблона и снапшота
- Пароль теперь устанавливается агентом через `lxc exec ... chpasswd` сразу после
  `lxc start` — и при создании, и после каждого `lxc restore`
- Снапшот теперь хранит чистую ОС без пароля, не зависит от ротации секретов

### v0.32 — технические исправления Ubuntu 24.04
- Исправлена настройка SSH: в Ubuntu 24.04 `sshd_config` использует
  `Include sshd_config.d/*.conf`, поэтому `sed` по основному файлу ненадёжен.
  Теперь создаётся `/etc/ssh/sshd_config.d/99-vpsemu.conf` с явными директивами
- Исправлен FALLBACK в `_get_root_pass`: функция возвращала строку `"FALLBACK"`
  которая присваивалась `ROOT_PASS` — теперь возвращает код `1` и агент
  останавливается для запроса пароля у пользователя
- Заменён `sleep 3` на `until systemctl is-system-running` в двух местах
  (создание и restore) — детерминированное ожидание готовности systemd
  вместо фиксированной задержки
### v0.33 — обнаружение форка сессии
- Добавлена проверка форка при инициализации: форк сессии в opencode копирует весь
  контекст родителя включая `AGENT_ID`, что привело бы к коллизии контейнеров
  у двух параллельных форков
- При загрузке скилла агент проверяет: есть ли контейнеры с текущим `AGENT_ID`
  которые он не создавал в этой сессии — если да, `AGENT_ID` регенерируется
- Безопасно по дизайну: контейнеры созданные в текущей сессии известны из контекста;
  незнакомые контейнеры указывают на форк → регенерировать
